Failures — require fixes before shipping
FAIL
Fish rate HUD understates real catch rate when mastery is active calcFishRate() · game.js ~line 3514

calcFishRate() computes displayed fish/s as sum((getSpeedMult() × typeMult × getPearlSpeedMult() × getMultiCatch()) / aDef.rate) — but it does not include getMasteryAutoSpeedMult().

The actual autoTick() speedBase (line ~2585) correctly multiplies getSpeedMult() × getPearlSpeedMult() × getMasteryAutoSpeedMult(). The Fishdex Mastery auto-speed bonus (up to ~+24% total across zones) is applied to real catches but invisible in the HUD counter. Players with high mastery will see a lower fish/s number than they are actually producing.

Fix: add getMasteryAutoSpeedMult() to the calcFishRate() product chain.
FAIL
renderHallOfFame() is dead code — never called, element ID wrong renderHallOfFame() · game.js ~line 4588

renderHallOfFame() queries document.getElementById('halloffame-content') which does not exist in index.html. The Hall of Fame screen uses id="stats-content" instead.

Additionally, renderHallOfFame() has zero call sites — the screen navigation for showScreen('halloffame') calls renderStatistics(), not this function. The trophy/zone/title rendering inside renderHallOfFame() is entirely inaccessible. Zone competition records in G.hofActive are partially shown in the stats tab 5 via _renderStatsContent(), not through this dedicated function.

Fix: wire renderHallOfFame() into showScreen('halloffame'), or merge its content into renderStatistics() and delete the dead function.
FAIL
Redeem code diamond reward displays wrong currency name redeemCode() success handler · game.js ~line 4963

When a redeem code grants diamonds (r.rewardType === 'diamonds'), the success message appends "Black Pearls!" instead of "Diamonds!". The diamonds are correctly credited to G.diamonds, but players see the wrong currency name in the confirmation toast.

Fix: change the string literal from 'Black Pearls!' to 'Diamonds!' in the diamonds reward branch of redeemCode().
FAIL
Competition duration still at 5-minute prototype value COMP_DURATION_MS · game.js ~line 410

const COMP_DURATION_MS = 5 * 60 * 1000; — the inline comment reads "5 min prototype (60 min production)". The production value has never been applied. Competition join screens also display "5 minutes."

A 5-minute competition window significantly reduces the appointment-gaming value of competitions and makes the Hall of Fame feel low-stakes. This was flagged in previous audits as a known gap but remains unresolved.

Fix: change to 60 * 60 * 1000 before shipping. Update any UI strings that display the duration.
Warnings — non-blocking but worth addressing
WARN
getEstimatedHourlyIncome() missing speed multipliers — seagull and competition rewards scale too low getEstimatedHourlyIncome() · game.js ~line 3527

getEstimatedHourlyIncome() computes sum(count / aDef.rate) using raw rates with no speed bonuses. The real catch rate in autoTick() multiplies by getSpeedMult() × getPearlSpeedMult() × getMasteryAutoSpeedMult().

This function is used for two payout calculations: seagull tap reward (1% of hourly income) and base competition reward via getCompBaseReward(). A player with maximum Speed, Pearl Speed, and Mastery bonuses (potentially 4–6× combined) receives seagull and competition rewards calculated as if they had no upgrades. The reward gap grows with progression.

Verify whether intentional (a cap mechanism) or a bug. If a bug: add getSpeedMult() × getPearlSpeedMult() × getMasteryAutoSpeedMult() to the rate denominator.
WARN
G.currentBobber missing from DEFAULT_STATE — no bobber equip mechanic exists DEFAULT_STATE · game.js

G.currentBobber is referenced in getBobberSrc() and reset in doPrestige(), but the field is absent from DEFAULT_STATE. New players start with G.currentBobber = undefined. getBobberSrc() gracefully falls back to the basic bobber image, so there is no crash — but the field is never written during normal gameplay. No "equip bobber" mechanic exists anywhere in the shop or inventory UI.

The bobber shop shows unlocked bobbers and upgrades their stats, but equipping is unimplemented. The bobber visual is always the default.

Either add G.currentBobber to DEFAULT_STATE and implement the equip flow, or remove the field references if the bobber visual system is deferred.
WARN
G.records (personal fishing records) tracked but never displayed updateRecord() · G.records · game.js

G.records is populated by updateRecord() to track best fish per zone per species. It is distinct from G.trophyRecords (heaviest per species) and the new G.stats.rec* fields (Records tab 7). The G.records data is never rendered anywhere in the current UI — it is written on every catch and saved to localStorage but is dead data from the player's perspective.

The Hall of Fame zone records section in renderStatistics() displays G.hofActive (competition wins), not G.records.

Either surface G.records in a UI panel, or remove the tracking to reduce storage writes.
WARN
rapid_waters and bite_speed_boost claim handlers double-call scheduleNextSpecialEvent() rapid_waters.claim(), bite_speed_boost.claim() · game.js ~line 5180, 5196

Both rapid_waters.claim() and bite_speed_boost.claim() explicitly call scheduleNextSpecialEvent() at the end of their handlers. closeSpecialEventPopup() — which runs after every claim — also calls scheduleNextSpecialEvent(). This means the next-event timer is started twice in sequence for these two events.

The last write wins, so the 15–30 min countdown is reset to a fresh value on the second call. Net effect: the timer works correctly but starts slightly later than intended. The other three events (care_package, lost_treasure, special_catch) delegate entirely to closeSpecialEventPopup() without this redundancy.

Remove the scheduleNextSpecialEvent() calls from rapid_waters.claim() and bite_speed_boost.claim().
WARN
fishdexPct divisor /180 — verify this matches actual reachable item count _renderStatsContent() tab 4 · game.js

Stats tab 4 computes completion as Math.round((G.fishdex.length / 180) * 100). Both autoTick and the manual btn-catch-ok handler push to G.fishdex, so in principle all 180 items (6 zones × 30) are reachable. However, 10 fish in FISH_DB are marked manual-only (automation cannot catch them). If those 10 manual-only fish are in G.fishdex scope, the total is still 180 and the divisor is correct.

Verify: confirm that G.fishdex entries are pushed for manual-only fish on catch, and that the complete reachable set is exactly 180. If the actual reachable set differs, the percentage cap will be wrong.

Audit FISH_DB count + all plant/trash entries vs. 180. Add a computed constant (FISHDEX_TOTAL) rather than the hardcoded literal so the number stays in sync automatically.
